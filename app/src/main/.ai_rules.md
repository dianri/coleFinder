# ColeFinder - Protocolo de Arquitectura y Desarrollo (Actualizado Marzo 2026)

## 1. Identidad y estado del proyecto
- **Propósito:** App nativa para docentes que geolocaliza centros educativos, con búsqueda por dirección, filtrado y distancia en tiempo real.
- **Estado actual:** MVP avanzado. UI con `SearchBar` (M3), mapa con marcadores numerados por cercanía, permisos GPS y deduplicación en app (`.distinctBy { it.id }`) además de limpieza y unicidad en base de datos donde aplique.
- **Backend:** Supabase / Postgres (PostgREST), alojado en Proxmox: `https://api.dianrie.com`.
- **Reglas canónicas ampliadas:** `.cursor/rules/data-supabase.mdc`, `architecture.mdc`, `ui-compose.mdc`.

## 2. Stack tecnológico principal
- **Lenguaje:** Kotlin 2.0.21 (Target SDK 36) con KSP.
- **UI:** Jetpack Compose + Material 3 (sin XML salvo petición explícita).
- **Mapas:** Google Maps SDK para Android (`maps-compose`).
- **Inyección:** Hilt (Dagger-Hilt).
- **Red:** Supabase SDK (`postgrest-kt` v3.0.3) + Ktor.
- **Ubicación:** FusedLocationProvider + Geocoder nativo (gratuito), debounce **300 ms** en autocompletado de direcciones.

## 3. Modelo de datos y tablas Supabase

### Tabla de lectura para la app: `public.colegios`
- Único origen PostgREST para listar/mapar centros (`postgrest["colegios"]`).
- **Cliente (anon / authenticated): solo lectura.** En base de datos: **RLS habilitado**; políticas `colegios_select_anon` y `colegios_select_authenticated` (`FOR SELECT USING (true)`); revocados **INSERT, UPDATE, DELETE, TRUNCATE** sobre `colegios` para `anon` y `authenticated`. La app **no** debe intentar escrituras salvo cambio explícito de producto y políticas.
- Población de datos: importación / SQL / Edge Functions / `service_role` en servidor — **no desde la UI.**

### Staging importación Madrid: `public.colegios_import_madrid`
- Tabla temporal para CSV del catálogo Madrid (columnas tipo `PK`, `NOMBRE`, `DESCRIPCION-ENTIDAD`, coordenadas, etc.).
- **No** es tabla de lectura de la app. Flujo: CSV **UTF-8** → staging → normalizar → volcar a `colegios`.
- Duplicados por origen: clave lógica **`fuente` + `fuente_id`**; en BD existe índice único `colegios_fuente_fuente_id_key` sobre `(fuente, fuente_id)`.

### Entidad Kotlin `Colegio` (`@Serializable`)
Mapeo **mínimo** que usa la UI hoy (PostgREST debe devolver estos campos o defaults coherentes):
- `id`: Int (PK en BD)
- `nombre`, `direccion`, `localidad`, `tipo`: String
- `latitud`, `longitud`: Double
- `telefono`: String?

En Postgres, `colegios` tiene columnas adicionales (Madrid: `fuente`, `fuente_id`, dirección desglosada, `categoria_madrid`, etc.). **No inventar** propiedades en Kotlin: ampliar el modelo solo si se consume en la app y el nombre coincide con la columna expuesta por PostgREST.

## 4. Estructura y arquitectura
- **Data:** `SupabaseClient` centralizado y `NetworkModule`.
- **MVVM / estado unidireccional:**
  - `MapState`: `isLoading`, `colegiosConDistancia`, `userLocation`, `filtroSeleccionado`, `cameraPositionState`, etc.
  - `MapViewModel`: fetch desde Supabase, Geocoder con debounce, ordenación por distancia.
- **Deduplicación:** antes de actualizar estado con listas de colegios: **`.distinctBy { it.id }`**.
- **Sin escrituras a Supabase desde composables;** capa ViewModel/datos y políticas acordadas.

## 5. Componentes de interfaz y UX (resumen)
- Buscador superior `SearchBar` (M3): Geocoder + sugerencias de colegios locales.
- Mapa: `myLocationButtonEnabled = false`, `compassEnabled = true`, `mapToolbarEnabled = false`; marcadores numerados por proximidad (hasta 50).
- FABs: mi ubicación y lista (bottom sheet).
- `ColegioDetailCard`: navegación y llamada vía Intents; cierre al tocar el mapa.

## 6. Reglas de desarrollo para la IA
- Kotlin moderno; Material 3; lógica de datos en el flujo de `MapState` / `MapViewModel`.
- Manejo de errores de red; sin trabajo pesado innecesario en el Main Thread.
- No duplicar acceso a Supabase en la UI.
- **Seguridad:** tablas `public` expuestas a PostgREST deben usar **RLS** y políticas explícitas; no exponer `service_role` en el cliente.

## 7. Próximos objetivos (roadmap)
1. **Persistencia (Room):** favoritos y modo offline.
2. **Sincronización:** actualización selectiva Room ↔ Supabase.
